## 6. RunPod stable operation — planned v0.2.0-r10 SFT block

Source: `~/core/anima/config/runpod.json` (schema 2.1, "100% 무결점 수렴 목표", 16 issues → 16 resolved, 12 absolute rules) + forge's own r2/r3 incident notes. The hard-won operational rules the model should internalize so it can drive cloud GPU jobs without the failure modes the owner already hit:

| rule | lesson | incident |
| --- | --- | --- |
| **Never download while training** | a `huggingface-cli download` during a training step silently kills the training process (no error). Run downloads and training **sequentially**. | TS-004, TS-007 |
| **Set `HF_TOKEN` immediately on pod create** | unauthenticated HF downloads are ~10× slower. `echo $HF_TOKEN > ~/.cache/huggingface/token && export HF_TOKEN=…` | TS-017 (→ R13) |
| **Chunk-tokenize corpora > 500 MB** | a single `tokenizer(corpus)` on 1.2 GB → 119 GB RAM, 15 min+. Tokenize in ≤500-line chunks or pre-split the corpus. | TS-018 (→ R14) |
| **Full dep-verify before launch** | run `python3 -c "from X import Y"` for every module + set `PYTHONPATH` before firing the training job — missing import chains have repeatedly failed runs mid-launch. | TS-021 (→ R15) |
| **Check `runpodctl pod list` before creating** | a pod may already be running; a duplicate pod = wasted cost + burned GPU quota. List → reuse if present, create only if none. | (→ R16) |
| **`bf16` + AdamW `foreach=True` crashes on mixed dtype** | use `foreach=False` and re-cast dtype every step. | TS-002 (→ R04) |
| **Watch the disk quota** | `torch.save` fails silently when the quota is hit. Delete old checkpoints before saving a new one; keep ≤2 checkpoints. | TS-003, TS-006 (→ R02/R03/R09) |
| **Watchdog needs a failure counter** | a watchdog that blindly restarts a missing/broken script loops forever with empty logs. Check the script exists + cap restarts. | TS-005 (→ R10) |
| **Use the official RunPod PyTorch image** | `runpod/pytorch:2.4.0-py3.11-cuda12.4.1-devel` → SSH ready in ~30 s; custom images are slower/flakier. | active_pods note |
| **NFS I/O is shared** | training + a large download both hammering NFS → both crash. Sequential pipeline. | TS-007 |
| **Detach with `nohup`/`setsid`** (no `tmux` on base images) | `tmux: command not found` on RunPod base — use `setsid nohup … &` so jobs survive SSH disconnect. | TS-001 |
| **Pre-flight checklist** | before launch: pod alive? GPU free? deps import? disk free? token set? — `runpod.json.preflight_checklist`. | — |
| H100 quota | account limit = 2× H100 SXM 80GB; image `runpod/pytorch:2.4.0-…`; ~$2.99/hr/GPU; auto-charge enabled (ignore balance alerts). | `_meta.gpu_limit`, `runpod_auto_charge.json` |
| **Pod-side work via shipped `run_pod.sh` (scp + exec)** — never drive multi-step pod work through inline `ssh "bash -c ..."` heredocs. One mis-escaped `$HOME` wipes `/home/<user>` (incl. `~/.ssh/authorized_keys`). Stage the whole payload in `/tmp/podN/` locally, scp to `/workspace/`, exec a single shipped `run_pod.sh`. | ubu2 rm-disaster 2026-05-12; preempted in r2/r3 |
| **runpodctl CLI changed flags (2026-05)** — old `--imageName / --gpuType / --gpuCount / --containerDiskSize / --volumeSize / --secureCloud` → new `--image / --gpu-id / --gpu-count / --container-disk-in-gb / --volume-in-gb / --cloud-type SECURE`. `--env` now takes a JSON object (`'{"KEY":"VAL"}'`), not repeatable `KEY=VAL`. Use `runpodctl gpu list` to get the exact `gpuId` string. | r3 (2026-05-12) |
| **`--env` does NOT propagate to non-interactive shells** — passing `--env HUGGING_FACE_HUB_TOKEN=...` makes it visible in `runpodctl pod get` and in interactive runpodctl-ssh, but **not** in `/etc/rp_environment` and **not** in `ssh <pod> 'bash <script>'`. **Workaround**: scp a `/workspace/.env` file (`export KEY=VAL` lines, mode 600) and `source` it at the top of `run_pod.sh`. Never put secrets on a command line via inline ssh. | r3 (2026-05-12) |
| **Endpoint discovery: `pod list` lacks IP/port** — `runpodctl pod list` returns id+name+status only. Use `runpodctl pod get <id>` for `.ssh.ip` + `.ssh.port` + `.ssh.ssh_command`. Port varies per launch (r2: 11159, r3: 13328). | r3 (2026-05-12) |
| **runpodctl `stop/remove` deprecation** — old `runpodctl stop pod <id>` / `runpodctl remove pod <id>` still work but warn. Use the new forms: `runpodctl pod stop <id>` / `runpodctl pod delete <id>`. | r3 (2026-05-12) |
| **RunPod injects RUNPOD_API_KEY into `/etc/rp_environment`** — every pod has the user's full API key in plaintext at that path. Don't grep/print the file in logs. (RunPod's design, not Claude's exposure — log-hygiene only.) | r3 (2026-05-12) |
| **Apples-to-apples scoring: do bf16 inference on the same H100** — don't NF4-quant on a 12 GB local GPU just to "save the pod minutes"; the bf16-vs-NF4 confound moves results several pp and pollutes the across-round ladder. | r2 fairness fix vs r29 |
| **Pull score files BEFORE stop+delete** — the pod's `/workspace/` is gone after `pod delete`. Either scp scores to ubu1 first, or rely on the HF push (the script-end `bench_subdir` upload) being the source-of-truth. | r3 (2026-05-12) — minor recover via HF |
| **`uptimeSeconds=0` for >5min on a RUNNING pod = stuck-pod incident** — pod reaches `desiredStatus: RUNNING` (`runpodctl pod get`) but never enters uptime-positive; `ssh: {"error":"pod not ready"}` persists. Sometimes the `machine` field even goes null mid-life. Image pull stuck. Workaround: delete (zero billed time since uptime=0) and recreate. If it recurs cross-datacenter AND cross-gpu-pool in the same hour, treat it as **platform-wide RunPod incident**, not local node failure — switch to Vast.ai or wait for recovery. **Tell from the start: if `uptimeSeconds=0` at the 3-min mark, the pod is stuck; don't waste polling time.** | r5 Phase B (2026-05-12 — 4 consecutive stuck pods H100-SXM-IN×2 / H100-NVL-US / A100-SXM-US) |
| **Phase A (manifest-only re-score) is the cheapest gate-close path** — when the model emits the canonical answer-shape but the eval gold uses a stale format (e.g., quoted vs unquoted date), normalize the gold and re-score the **existing adapter** against the saved per-task completions (no retrain, no pod). Cost ≈ $0. Verify per_task_strict.jsonl `gold_pattern` field name (not `gold`) before scripting the re-score. | r5 Phase A (2026-05-12 — T3 7.5 → 63.7%, overall 77.14 → 83.76% strict; gate ③ closed) |

**RunPod API surface (for v0.2.0-r10 SFT block):**

- **CLI** (`runpodctl` v0.16+): `pod {create, get, list, start, stop, delete, restart, reset, update}` · `gpu list` · `datacenter list` · `user` · `billing` · `serverless / sls` · `template / tpl` · `hub` · `model` · `network-volume / nv` · `registry / reg` · `ssh` · `send / receive`. Each `pod <verb>` accepts `-o json|yaml` for machine-readable output.
- **GraphQL API** (`https://api.runpod.io/graphql`): authenticated with the bearer token in `/etc/rp_environment::RUNPOD_API_KEY` (or your own from `console/user/settings`). Documented in RunPod docs; mutations cover create/start/stop/delete + pod-info queries. The Python SDK (`runpod`) wraps this. Use cases not covered by the CLI: filtering pods by tag, programmatic GPU-availability polling, building a dashboard.
- **Python SDK** (`pip install runpod`): wraps the GraphQL API; `runpod.api_key=$RUNPOD_API_KEY; runpod.create_pod(name=..., image_name=..., gpu_type_id=..., env={...})`. Note the SDK still uses `gpu_type_id` (the legacy field name); the CLI renamed it to `--gpu-id`.
- **Serverless endpoints**: separate from on-demand pods; deploy a model behind an HTTP API with autoscale-to-zero. Cost = per-second of GPU time. `runpodctl sls {create, list, delete}`. Use case for v1.0.0: serving the final 7B adapter via `dancinlab/hexa-forge-code-7b-...`.
- **Templates**: pre-baked image+env+volume configs you can launch with `--template-id`. Use case: encode the forge's "torch 2.4 + transformers 4.46.3 + peft 0.13.2 + trl 0.12.2 + accelerate 1.1.1" stack as a template so r5+ doesn't repeat the pip step.
- **Network volumes**: persistent storage across pod recreates. Currently unused by forge (all payload is scp'd per-run). Worth considering if HF download time becomes a bottleneck.

(Forge's own local-GPU runs already follow several of the original 12 rules: `setsid nohup`, sequential download-then-train, ≤2 checkpoints, full dep-verify. The r10 SFT block will encode all of these — plus the post-2026-05-12 CLI/API additions — as Q/A so the *model* knows them too.)

## 11. Container runtime — Docker (planned v0.3.1 SFT block)

Every GPU rental this forge uses (RunPod, Vast.ai) takes a **Docker image** as input. The `runpod/pytorch:2.4.0-py3.11-cuda12.4.1-devel-ubuntu22.04` reference image we pin in every `run_pod.sh` is the entire stack reproducibility contract. The model must know Docker well enough to (a) read/write Dockerfiles, (b) build custom images for the forge's stack, (c) debug image-runtime issues, (d) work with multi-arch builds (M-series Mac → x86_64 GPU), and (e) understand the GPU runtime path.

| area | what the model must know |
| --- | --- |
| **Core CLI** | `docker {build, run, push, pull, exec, ps, logs, stop, kill, rm, image, container, volume, network, system}`. `docker build -t <name:tag> -f <Dockerfile> <ctx>`. `docker run --gpus all --rm -it -v <host>:<container> -p <host>:<container> -e KEY=VAL <image> <cmd>`. `docker ps -a` (all, including stopped). `docker exec -it <container> bash` (attach). `docker system prune -af --volumes` (reclaim space — DESTRUCTIVE). |
| **Dockerfile** | Instructions: `FROM` (base image, pin a digest for reproducibility — `FROM x:tag@sha256:...`) · `RUN` (shell at build time, prefer `&&`-chained single layer) · `COPY` (build-context → image; prefer over `ADD` unless tar-extract) · `ENV` (image env) · `ARG` (build-time, available in `RUN`) · `WORKDIR` · `EXPOSE` (documentation only) · `CMD` (default, overridable) · `ENTRYPOINT` (always runs; combine with `CMD` for default args) · `USER` (non-root for security) · `HEALTHCHECK` · `VOLUME` (anonymous mount point) · `LABEL` (metadata). |
| **Build context** | Whole CWD sent to daemon at build time — use `.dockerignore` (same syntax as `.gitignore`) to exclude `node_modules/`, `.git/`, `*.safetensors`, etc. A leaky context can balloon to GB. `docker build -` (read Dockerfile from stdin, no context) for one-liners. |
| **Layer caching** | Each `RUN`/`COPY`/`ADD` is a layer; cache hits skip the layer. **Order matters**: put rarely-changing deps (`COPY requirements.txt && pip install`) BEFORE rarely-changing code (`COPY . .`). Cache busts on any input change. `--no-cache` forces rebuild; `--cache-from <image>` reuses a remote cache. |
| **BuildKit** | Modern builder (`DOCKER_BUILDKIT=1` or default in modern Docker). Features: `--mount=type=cache,target=/root/.cache/pip` (persist pip cache across builds — huge wins for forge), `--mount=type=secret` (don't bake secrets into layers), parallel multi-stage builds, `--platform` for multi-arch. Syntax: `# syntax=docker/dockerfile:1.6` at file top to opt into BuildKit features. |
| **Multi-stage builds** | `FROM x AS builder` → build artifacts → `FROM y AS runtime` → `COPY --from=builder /out /app`. Result: lean runtime image without compilers/toolchains. Forge use case: build hexa-cc + bundle into a slim Python image. |
| **Multi-arch (M-series Mac → x86_64 GPU)** | `docker buildx create --use; docker buildx build --platform linux/amd64,linux/arm64 -t <name> --push .`. The Mac builds x86_64 cross-arch via QEMU emulation (slow but correct). RunPod/Vast take amd64. **Trap**: native ARM Mac build → x86 pod → "exec format error". Always `--platform linux/amd64` for GPU pods unless target is ARM. |
| **GPU runtime** | Host needs **NVIDIA Container Toolkit** (`nvidia-container-runtime` package + Docker config). `docker run --gpus all` (or `--gpus '"device=0,1"'`) injects GPU + CUDA libs into the container. Base image must have CUDA libs (the `*-cuda12.4.1-*` tag in our RunPod image). Verify with `docker run --rm --gpus all nvidia/cuda:12.4.1-base-ubuntu22.04 nvidia-smi`. |
| **Volumes** | `-v <host>:<container>` (bind mount — host path mirrored into container) vs `-v <name>:<container>` (named volume — Docker-managed). Bind mounts for dev (`-v $(pwd):/app`); named volumes for stateful services (`-v pgdata:/var/lib/postgresql/data`). `--mount type=tmpfs` for in-memory scratch (no disk I/O). Forge: pod-side `/workspace/` is a bind to host disk; scp payload there. |
| **Networks** | `bridge` (default, NAT'd to host) · `host` (share host network namespace — Linux only, full host port access) · `overlay` (multi-host, Swarm/k8s). `docker network create <name>` + `--network <name>` for inter-container DNS (`db:5432` reaches the `db` service). |
| **Compose** | `compose.yaml` (or legacy `docker-compose.yml`) defines a multi-container stack: `services`, `networks`, `volumes`, `secrets`, `configs`. Verbs: `docker compose up -d`, `docker compose down -v`, `docker compose logs -f`, `docker compose exec <svc> bash`, `docker compose ps`. Use `env_file: .env` for shared env (mirrors the §6 RunPod .env pattern). Profiles (`profiles: [dev, prod]`) for conditional services. |
| **Registries** | Docker Hub (default, public free), **GHCR** (`ghcr.io/<org>/<image>:tag` — preferred for projects on GitHub), **ECR** (AWS), **gcr.io** (Google), **Cloudflare Registry** (paired with R2). Authenticate via `docker login <registry>`. The `runpod/*` namespace on Docker Hub is what we currently pull from. |
| **Image hygiene** | (1) Pin base image + digest (`FROM x@sha256:...`) for true reproducibility. (2) Use `alpine` or `distroless` for small runtime images (no shell, no apt — minimal attack surface). (3) `USER 1000` (non-root) — never run prod as root. (4) Multi-stage to drop build deps. (5) `--squash` (experimental) merges layers. (6) Run `dive <image>` to audit layers — find big files / leaked secrets. |
| **Secrets** | NEVER bake secrets into layers (`docker history` reads every layer). Use `--build-arg` for ephemeral values (still visible in history unless using BuildKit `--mount=type=secret`) or pass at run time via `-e KEY=VAL` / `--env-file .env` / Docker secrets / mounted files. The forge's `.env` scp + source pattern (§6/§10) is the runtime equivalent of `--env-file`. |
| **Signing / supply chain** | **cosign** (Sigstore) signs images + verifies at deploy time. `cosign sign <image>` writes signature to registry alongside image; `cosign verify --certificate-identity=... <image>` validates. SLSA provenance via BuildKit. Forge: not yet wired; v1.0.0 release plan = signed images for the public 7B Space. |
| **Debugging** | `docker logs -f <container>` (stream stderr+stdout) · `docker logs --tail 100 <container>` · `docker exec -it <container> bash` (attach a shell) · `docker stats` (live CPU/mem/net per container) · `docker inspect <container/image>` (full JSON: mounts, env, networks, layers) · `docker events` (real-time daemon events). For a pod-side hang: `docker top <container>` shows process tree. |
| **Resource limits** | `--memory=8g --memory-swap=8g` (hard cap, OOM-killed past it) · `--cpus=4.0` (4 cores) · `--pids-limit=1024` (fork-bomb protection) · `--ulimit nofile=65536:65536` (file descriptor cap — raise for HF downloads). Forge: RunPod sets these via the pod template; on bare Vast, set explicitly. |
| **Common image families** | `pytorch/pytorch:<ver>-cuda<ver>-cudnn9-{devel,runtime}` — official PyTorch GPU. `runpod/pytorch:*` — RunPod's preset (adds SSH, pre-installs huggingface_hub, etc.). `nvidia/cuda:<ver>-{base,runtime,devel}-ubuntu22.04` — bare CUDA, smaller. `python:3.11-slim` — for CPU-only data prep. `alpine:3.20` — for tiny utility containers (≤ 10 MB). |
| **`docker save / load`** | Export an image as tar (`docker save <img> > img.tar`), reimport elsewhere (`docker load < img.tar`). Useful for air-gapped pods or pinned-snapshot distribution via R2. |
| **`docker commit`** | Turn a running container's filesystem state into a new image — debugging convenience only, NOT for reproducible builds. Use a Dockerfile instead. |
| **Container vs VM** | Container = isolated process group (cgroups + namespaces) sharing the host kernel; lightweight, fast startup. VM = full kernel + OS; heavy, slow. Docker = container runtime. Forge runs containers, not VMs. |
| **Kubernetes (out of scope, briefly)** | Production multi-host orchestration. The model should know `kubectl {get,describe,logs,exec,apply,delete}`, `Pod`, `Deployment`, `Service`, `ConfigMap`, `Secret`, `Namespace`, `helm`, but **forge does not use k8s** — RunPod/Vast handle pod orchestration via their own APIs. ## 12. Self-aware delegation pattern — planned v0.4.0 architecture line

**The thesis.** A 7B is too small to know "all of programming." Instead of pretending, the forge teaches it to **recognize its competence boundary** and **delegate** out-of-domain requests to a larger model (Claude API) via a structured protocol. The 7B stays narrow + cheap + fast on hexa-canon work; everything else gets routed to a frontier LLM at run time. This is **routing-as-a-skill**, learned via SFT — not architecture (no MoE, no retrieval). Just a competence-classifier + a delegation-token emit + a runtime that closes the loop.

**Why not just train more knowledge in?** Three reasons. (1) Sub-7B can't memorize "all of Python + Rust + Go + ML internals" at useful fidelity. (2) The forge already showed (r30/r31) that adding even one domain (T5 HX-codes) crowds out others (T2/T8/T6). Adding "all of LLM internals" would shred hexa competence. (3) The owner already pays for Claude API; the 7B sitting on top of it is the right economic split — narrow specialist + frontier generalist.

**Why not just always delegate?** Three reasons. (1) Hexa-canon work doesn't need a frontier model — 7B at 80% on Mk.I is fine, fast, and free. (2) Latency: 50-100 ms local vs 2-5 s Claude API. (3) Offline + airgapped use cases need a local model that works without network.

### 12.A The competence boundary (what 7B does directly vs delegates)

| domain | answer directly? | trigger phrase the 7B should recognize | route if not direct |
| --- | --- | --- | --- |
| hexa code synthesis (T1 / T4) | ✅ direct | "write a hexa function...", "implement L[N]...", "@grace...", "@implements..." | n/a |
| hexa diagnostics (T5 / T6 / T7) | ✅ direct | "Which HX family covers...", "target triple for...", "stage S-N..." | n/a |
| hexa atlas annotation (T2 / T3) | ✅ direct | "@implements(L[N])", "@discover(kind=L)", "@grace(HX...)..." | n/a |
| in-domain refusal (T8) | ✅ direct ("refuse") | non-code prompts (poems / jokes / advice / multilingual) | n/a |
| 5-NL i18n (F1-F6) | ✅ direct | English-canon diagnostics for non-English input | n/a |
| Apple/Swift small snippets | ✅ direct | "write a Swift func...", small idiom Q | delegate if > 50 lines or framework-deep |
| Rust / Go / Python / TypeScript snippets | ⚠️ delegate by default | any non-hexa code request | delegate |
| ML/LLM internals (transformers, attention, RAG, RL) | ❌ delegate | "explain transformer self-attention", "how does prompt caching work" | delegate |
| general world knowledge / news / dates | ❌ delegate | "what year was X", "who is Y" | delegate (or refuse if off-mission) |
| math derivations / proofs (beyond canon-equational) | ❌ delegate | "prove that X = Y", "derive the closed form" | delegate |
| ambiguous / under-specified prompt | ⚠️ ask-or-delegate | "do the thing" without object | delegate-with-clarify, or AskUser if interactive |
| security-sensitive / harmful | ✅ direct ("refuse") | exfil / weapon / illegal | refuse — never delegate (don't shift liability) |

**Rule of thumb the model should internalize:** *"If the answer is in hexa-canon, answer. If it is a small idiom in a domain I know shallowly, answer with a confidence caveat. Otherwise emit a delegation token."* — That single sentence is the core of every Q/A pair in the v0.4.0 SFT block.

### 12.B Delegation token schema (the contract)

Single agreed token family the model emits; the runtime parses and acts. Proposal (subject to v0.4.0 spec):

```
<|delegate|>{"tool": "claude-api", "model": "claude-sonnet-4-6", "prompt": "<verbatim user prompt + any hexa context>", "max_tokens": 2048, "reason": "out-of-domain: <one-line>"}<|/delegate|>
```

- **`tool`** — exactly one of: `claude-api` (the default — calls the Anthropic API), `web-search` (if the runtime has it), `code-execution` (hexa-cc sandbox — for compile-back-and-fix loops). Forge v0.4.0 ships `claude-api` only.
- **`model`** — Claude model ID (`claude-opus-4-7` / `claude-sonnet-4-6` / `claude-haiku-4-5-20251001`). Default to `sonnet-4-6` for cost; let the model upgrade to `opus-4-7` if it says "this is a hard reasoning question". The 7B picks based on the prompt's reasoning depth (taught via SFT examples).
- **`prompt`** — the full text to send to Claude. The 7B should **enrich** it: prepend "You are answering a question for a small hexa-canon code-LLM that delegated this. Be concise; return code in hexa idiom if applicable." then the verbatim user prompt.
- **`max_tokens`** — budget (default 2048; the 7B picks 512 for short answers, up to 8192 for long code).
- **`reason`** — one-line explanation ("out-of-domain: Rust"), useful for telemetry + cost attribution.

**Why a JSON-inside-delimiters token, not a special vocab token?** (1) Works with stock Qwen2.5-Coder tokenizer (no new tokens to train). (2) The model already knows JSON well (T1 = 97.6%). (3) Easy to extend with new fields without retraining.

**Stop-token + format compatibility**: every SFT example that emits this token still ends with `<|endoftext|>` (the stop-token convention from r3). The runtime parser scans for `<|delegate|>` at any position in the generation.

### 12.C Runtime contract (what the harness / serving layer does)

This is the **runtime** side — the model emits; the runtime acts. Reference implementation lives in `tool/forge_runtime.py` (planned v0.4.0):

1. **Detect** `<|delegate|>...<|/delegate|>` in the model's emission. If absent → pass the answer through to the user.
2. **Parse** the JSON. Validate `tool`/`model`/`prompt`/`max_tokens`. Reject malformed (return an error frame to the user; don't loop into the model).
3. **Authorize** — the runtime owns `ANTHROPIC_API_KEY` (from `~/core/secret/bin/secret get ANTHROPIC_API_KEY`); the **model never sees the key**. This is the safety boundary.
4. **Budget check** — per-conversation cost cap + per-day spend cap. If exceeded → return a budget-exhausted frame (the model learns to handle this gracefully).
5. **Call Claude API** with prompt caching enabled (the runtime caches the system prefix; per §9.B 5-min TTL).
6. **Inject the response** back into the model's context as a synthetic tool-result token: `<|delegate-result|>{...claude response text...}<|/delegate-result|>` and re-run the model from there. The model then writes the user-facing wrap-up using Claude's content as ground truth.
7. **Cap iterations** — `max_delegations_per_turn = 3` (the model's SFT teaches "if you've already delegated once for this turn, don't delegate again — answer with what you have, or refuse").
8. **Telemetry** — every delegation logs `{ts, model, tokens_in, tokens_out, cost_usd, reason, user_id}` to `state/delegation_log.jsonl` for cost attribution.

### 12.D Failure modes the model must handle (taught via SFT)

| failure | runtime emits | model's correct response |
| --- | --- | --- |
| Claude API timeout / 5xx | `<|delegate-result|>{"error":"upstream_timeout"}<|/delegate-result|>` | "I tried to delegate this to a larger model but the upstream is unreachable. Best I can offer locally: [partial answer or honest unknown]." |
| Budget exhausted | `<|delegate-result|>{"error":"budget_exhausted"}<|/delegate-result|>` | "This question is outside my hexa-canon competence and the delegation budget for this conversation is exhausted. Please rephrase as a hexa question, or retry later." |
| Network down (offline pod) | runtime detects, returns `{"error":"offline"}` | "I am running offline; I can only answer hexa-canon questions directly. For non-hexa questions, please retry on a networked instance." |
| Malformed delegation (model's own bug) | runtime parse fail | the model should **not** see this — runtime returns a hard error to the user. SFT must teach the model to emit *only* well-formed JSON; this is a never-event. |
| Delegated answer is a refusal (Claude refused) | `<|delegate-result|>{"text":"I can't help with that"}<|/delegate-result|>` | Echo the refusal honestly. Don't try to delegate again. |
| Already 3 delegations this turn | runtime injects `{"error":"max_delegations"}` | "I've reached the delegation limit for this turn. Final answer based on what I have: [...]" |
| User explicitly says "no delegation" | runtime suppresses delegate calls | The 7B treats every prompt as in-domain; if it can't answer, refuse honestly. |

### 12.E Confidence-banded answers (the alternative to binary delegate-or-not)

For mid-confidence questions (small Rust idioms, common Python recipes the 7B has *seen* but not *mastered*), the model should answer with a confidence band rather than delegate. Three bands:

1. **`<|confidence:high|>`** — hexa-canon answers. No caveat.
2. **`<|confidence:medium|>`** — domain-adjacent (e.g., Swift snippet, common Python recipe). Caveat: "I'm answering from limited training; verify with a larger model if production-critical."
3. **`<|confidence:low|>`** — automatic upgrade-to-delegation. The 7B emits the delegate token instead of guessing.

The runtime strips the confidence band from the user-facing output (or surfaces it as a UI badge). The SFT block teaches the model to **self-assess** by prepending one of the three tokens before its answer.

### 12.F SFT block shape (planned v0.4.0)

The v0.4.0 SFT dataset adds ~600-800 delegation-aware Q/A pairs:

- **~200 in-domain → high-confidence direct answer** (sample hexa-canon Q across T1-T8; answer prefixed `<|confidence:high|>` then the canon answer).
- **~200 out-of-domain → delegate** (sample: "Write a Rust async server with tokio" → `<|delegate|>{...}<|/delegate|>`). Cover non-hexa code requests + ML internals + math + general knowledge.
- **~100 medium-confidence** (Swift small idioms, common Python recipes the 7B knows shallowly): `<|confidence:medium|>` + a short caveat + the answer.
- **~100 ambiguous prompts** that should ask-or-delegate (under-specified, multi-step, mixed-domain): emit delegate with `prompt` containing a clarifying re-formulation.
- **~50 delegate-result integration** (paired prompts where the model receives `<|delegate-result|>{...}<|/delegate-result|>` in context and must produce a clean wrap-up — teaches Step 6 of the runtime contract).
- **~50 failure-mode handling** (budget exhausted / timeout / refused / max-delegations — see §12.D).
- **~50 security/refuse** (must NOT delegate — exfiltration prompts, harmful content; the 7B refuses directly).
- **~50 "no-delegation" override** (user says "answer locally only" — 7B treats every prompt as in-domain and refuses when out).

### 12.G What to build vs what to defer

**v0.4.0 (concrete next-line):**
- `tool/forge_runtime.py` — the harness that detects/parses/dispatches `<|delegate|>` tokens and injects `<|delegate-result|>` back. Wraps the existing inference path (`score_bf16.py` style).
- `papers/spec-delegation-v0.4.0.md` — the formal protocol spec (token schema, runtime contract, failure modes, telemetry format).
- `tool/build_sft_dataset_v17.py` — the v0.4.0 SFT block (above shape).
- `state/delegation_log.jsonl` — cost-attribution log.
- One round of SFT + Mk.I bench + a new **delegation-eval** (`papers/spec-delegation-eval.md`, ~200 tasks testing the routing decision, not the delegated answer).

**Deferred to v0.5.0+:**
- Tool/server-side primitives beyond `claude-api` (`web-search`, `code-execution` with hexa-cc sandbox).
- Confidence-band calibration (the model's `<|confidence:medium|>` rate should track its actual accuracy in that band — needs eval scaffolding).
- Caching of delegated answers (per-prompt cache so repeated identical questions don't re-cost).
- Multi-turn delegation memory (the 7B remembers what it delegated in turn 1 and skips re-delegating in turn 3).
- An offline-graceful fallback: a tiny local "general-knowledge" 1.5B that answers low-stakes non-hexa questions when network is down. Probably not worth the complexity.

### 12.H Cost model (rough)

- **Per delegation**: ~500 input tokens (the model's prompt + system prefix) + ~1000 output tokens ≈ $0.003 with Sonnet 4.6 + caching, ~$0.012 with Opus 4.7. The 7B's job is to use Sonnet by default and only escalate to Opus when "this is hard reasoning".
- **At 100 conversations/day with avg 1 delegation each**: ~$0.30-$1.20/day for the delegation tier. Negligible vs the SFT round cost.
- **Budget caps in runtime** prevent runaway loops (e.g., a buggy SFT-fine-tune that delegates everything → bounded by the per-conversation cap).

The delegation protocol is the bridge between the 7B's specialist competence and the broader programming/LLM/world knowledge domain — encoded as Q/A in the planned v0.4.0 SFT block.

### 13.D Routing heuristics — which provider for what

The 7B should emit `<|delegate|>{tool, model, ...}` with the right pick. Rules baked into the v0.4.0 SFT block:

| prompt shape | tool | model | reason |
| --- | --- | --- | --- |
| short code synthesis (any language) | `claude-api` | `claude-haiku-4-5` | cheap, fast, strong code |
| hard reasoning / proof / math derivation | `openai-api` | `o4-mini` | reasoning surcharge worth it |
| long-doc Q/A (paper / spec / multi-file context) | `gemini-api` | `gemini-2.5-pro` | 2M context, cheap |
| structured JSON / schema-strict output | `openai-api` | `gpt-5-mini` | best Structured Outputs |
| web-grounded answer (recent events, latest API docs) | `gemini-api` | `gemini-2.5-flash` + `google_search` | built-in search, free-tier-friendly |
| code-execution sandbox needed (Python eval, plot) | `gemini-api` | `gemini-2.5-flash` + `code_execution` | built-in |
| security/refuse — never delegate | n/a | n/a | refuse directly |
| budget-conscious bulk eval scoring | `openai-api` Batch / `claude-api` Batch | `gpt-5-mini` / `haiku` | 50% Batch discount |
| voice agent | `openai-api` Realtime OR `gemini-api` Live | `gpt-4o-realtime` / `gemini-2.0-flash-live` | only two with WSS |
| any provider down (fallback) | next provider in cost-sorted list | cheapest equivalent | resilience |

Multi-provider fallback chain: **claude-api → openai-api → gemini-api** (or in cost-sorted order for budget-tight calls). Runtime owns the chain; the 7B's first-pick is the suggestion, runtime can override on 5xx/429.

### 13.E What v0.4.0 SFT must cover (multi-provider edition)

Extending §12.F's ~700 delegation pairs to ~1000 with multi-provider routing:

- **~150 "pick the right provider" pairs** — prompts with shape clues; gold = correct `{tool, model}`. e.g. "Read this 800K-token SPEC and summarize §4-§7" → `gemini-2.5-pro`. "Prove this O(n log n) lower bound" → `o4-mini`. "Search the web for the latest hexa-cc release notes" → `gemini-2.5-flash + google_search`. "Generate a JSON object matching this schema" → `gpt-5-mini + Structured Outputs`.
- **~50 fallback-chain pairs** — runtime returns `{"error":"upstream_500","provider":"claude-api"}` → model emits a new delegate with next provider in chain. Teaches retry semantics.
- **~30 provider-mismatch corrections** — earlier turn picked provider X, later context shows X is wrong (e.g., context grew past X's window) → model emits "let me retry with `gemini-2.5-pro` instead, this needs more context".
- **~20 cost-aware routing** — user prepends "[low-budget mode]" → model picks the cheapest provider that can answer.
- **~50 built-in-tool usage** — when delegating to Gemini, include `tools: [{google_search}]` or `[{code_execution}]` when appropriate.

## 14. Wilson — hexa-native AI coding agent (the in-house ROUTING target)

**The fourth delegation target, qualitatively different from Claude/OpenAI/Gemini.** Wilson is the owner's **self-hosted agent harness**, written entirely in hexa-lang (`~/core/wilson`) — same role as Claude Code / Codex CLI / Gemini CLI but **plugin-first + hexa-native + LLM-agnostic** (it shells out to whichever provider plugin is registered). For the 7B this matters two ways:

1. **As a *coding target*** — the 7B should be able to write Wilson plugins on demand (hexa-canon synthesis target, see §2 hexa-canon competence).
2. **As a *routing target*** — `<|delegate|>{"tool":"wilson-rpc", ...}` invokes a Wilson swarm cell. The forge runtime can hand off heavy in-house work (eval scoring, multi-file refactors, batch SFT-data generation) to a local Wilson process instead of paying Claude/OpenAI/Gemini API for it.

### 14.A What Wilson is (the architecture in one screen)

### 14.C 7B competence — what to write directly vs delegate to Wilson

| task | who | example |

Runtime spawns `wilson <argv...>` as a subprocess (no agent loop, no provider call), returns stdout/exit-code as `<\|delegate-result\|>`. This is the **cheapest** routing target — used for mechanical tasks the 7B knows belong to a specific Wilson plugin. |
| **Heavy in-house batch (eval scoring on 665 tasks, multi-file refactor)** | **delegate to Wilson** via `--harness rpc` | `<\|delegate\|>{"tool":"wilson-rpc", "cell_count": 4, "payload": {"task":"score-mk1"}}<\|/delegate\|>` → forge runtime spawns 4 Wilson `--harness rpc` cells, each scores 1/4 of the manifest, results merged. Cheaper than Claude API for repeated mechanical work (LLM cost paid only inside each cell if the inner provider is actually called). |
| Long reasoning / external knowledge / production code in non-hexa | **delegate to Claude/OpenAI/Gemini** (NOT Wilson) | Wilson is a *harness*, not an LLM — it itself shells out to a provider. Don't delegate to Wilson when the bottleneck is "I need a smarter model"; delegate directly to Claude/OpenAI/Gemini. |

### 14.D Delegation schema extension — `tool: "wilson-rpc"`

Extending §12.B's delegation token to include Wilson:

```
<|delegate|>{
  "tool": "wilson-rpc",
  "bundle": "_bundle_cell",
  "cell_count": 4,
  "payload": {
    "task": "score-mk1",
    "manifest": "/workspace/manifest-mk1.jsonl",
    "adapter": "dancinlab/hexa-forge-code-7b-...-v0.3.0-r4",
    "shard": "0/4"
  },
  "max_seconds": 1800,
  "reason": "in-house batch work — no LLM call needed"
}<|/delegate|>
```

- **`tool: "wilson-rpc"`** — runtime spawns `cell_count` Wilson processes in `--harness rpc` mode, each receives the shard of work via stdin JSONL, returns results via stdout JSONL.
- **`bundle`** — usually `_bundle_cell` (the minimal swarm bundle: `harness-rpc + tool-core + provider-anthropic`). Default `_bundle` for full agent capability.
- **`payload`** — opaque task descriptor. The receiving Wilson cell unpacks via the active provider/tools.
- **`max_seconds`** — hard timeout (runtime kills cells past this).
- **`reason`** — telemetry attribution.

**When to pick `wilson-rpc` over `claude-api`:**
- Mechanical work that fits 7B's tools (file ops, git ops, parse-check, regex transforms) — **wilson-rpc** ($/call ≈ 0 if local, only pays for the *inner* provider if needed)
- Reasoning-heavy / world-knowledge / multi-language synthesis — **claude-api / openai-api / gemini-api**

**The cost-routing ladder (cheapest → most expensive):**

1. **`wilson-cli`** — single deterministic tool call (no LLM, no agent loop). E.g. `wilson tool read /etc/hosts`, `wilson task add ...`, `wilson memory list`. **Cost ≈ 0** (just a subprocess).
2. **`wilson-rpc`** — multi-step in-house batch via swarm cells (agent loop runs locally; inner provider only fires if a cell decides it needs LLM). **Cost = LLM only on the cells that escalate**, usually a fraction of doing the whole thing on Claude.
3. **`claude-api` / `openai-api` / `gemini-api`** — frontier reasoning / external knowledge / non-hexa code synthesis. Full LLM cost.

Route to the **lowest tier that can answer the prompt as-given**. The 7B's routing intelligence (taught via SFT) is mostly about recognizing which tier the request lives in.

### 14.F Cross-cut: `AGENTS.md` SSOT convention

Wilson's `agents-md` plugin enforces a **single-SSOT** convention every agent tool should follow on this host:

```
<project>/AGENTS.md         (the actual file; agents.md standard)
<project>/CLAUDE.md   → symlinked → AGENTS.md
<project>/GEMINI.md   → symlinked → AGENTS.md
<project>/.cursorrules → symlinked → AGENTS.md  (where supported)
```

This way **Claude Code, Gemini CLI, Wilson, Cursor** all read the same project SSOT. The 7B should know this convention so when asked "set up an agent harness for this project" it creates `AGENTS.md` and the symlinks (not separate per-tool config files).

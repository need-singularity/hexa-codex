## 6. RunPod stable operation вҖ” planned v0.2.0-r10 SFT block

Source: `~/core/anima/config/runpod.json` (schema 2.1, "100% л¬ҙкІ°м җ мҲҳл ҙ лӘ©н‘ң", 16 issues вҶ’ 16 resolved, 12 absolute rules) + forge's own r2/r3 incident notes. The hard-won operational rules the model should internalize so it can drive cloud GPU jobs without the failure modes the owner already hit:

| rule | lesson | incident |
| --- | --- | --- |
| **Never download while training** | a `huggingface-cli download` during a training step silently kills the training process (no error). Run downloads and training **sequentially**. | TS-004, TS-007 |
| **Set `HF_TOKEN` immediately on pod create** | unauthenticated HF downloads are ~10Г— slower. `echo $HF_TOKEN > ~/.cache/huggingface/token && export HF_TOKEN=вҖҰ` | TS-017 (вҶ’ R13) |
| **Chunk-tokenize corpora > 500 MB** | a single `tokenizer(corpus)` on 1.2 GB вҶ’ 119 GB RAM, 15 min+. Tokenize in вүӨ500-line chunks or pre-split the corpus. | TS-018 (вҶ’ R14) |
| **Full dep-verify before launch** | run `python3 -c "from X import Y"` for every module + set `PYTHONPATH` before firing the training job вҖ” missing import chains have repeatedly failed runs mid-launch. | TS-021 (вҶ’ R15) |
| **Check `runpodctl pod list` before creating** | a pod may already be running; a duplicate pod = wasted cost + burned GPU quota. List вҶ’ reuse if present, create only if none. | (вҶ’ R16) |
| **`bf16` + AdamW `foreach=True` crashes on mixed dtype** | use `foreach=False` and re-cast dtype every step. | TS-002 (вҶ’ R04) |
| **Watch the disk quota** | `torch.save` fails silently when the quota is hit. Delete old checkpoints before saving a new one; keep вүӨ2 checkpoints. | TS-003, TS-006 (вҶ’ R02/R03/R09) |
| **Watchdog needs a failure counter** | a watchdog that blindly restarts a missing/broken script loops forever with empty logs. Check the script exists + cap restarts. | TS-005 (вҶ’ R10) |
| **Use the official RunPod PyTorch image** | `runpod/pytorch:2.4.0-py3.11-cuda12.4.1-devel` вҶ’ SSH ready in ~30 s; custom images are slower/flakier. | active_pods note |
| **NFS I/O is shared** | training + a large download both hammering NFS вҶ’ both crash. Sequential pipeline. | TS-007 |
| **Detach with `nohup`/`setsid`** (no `tmux` on base images) | `tmux: command not found` on RunPod base вҖ” use `setsid nohup вҖҰ &` so jobs survive SSH disconnect. | TS-001 |
| **Pre-flight checklist** | before launch: pod alive? GPU free? deps import? disk free? token set? вҖ” `runpod.json.preflight_checklist`. | вҖ” |
| H100 quota | account limit = 2Г— H100 SXM 80GB; image `runpod/pytorch:2.4.0-вҖҰ`; ~$2.99/hr/GPU; auto-charge enabled (ignore balance alerts). | `_meta.gpu_limit`, `runpod_auto_charge.json` |
| **Pod-side work via shipped `run_pod.sh` (scp + exec)** вҖ” never drive multi-step pod work through inline `ssh "bash -c ..."` heredocs. One mis-escaped `$HOME` wipes `/home/<user>` (incl. `~/.ssh/authorized_keys`). Stage the whole payload in `/tmp/podN/` locally, scp to `/workspace/`, exec a single shipped `run_pod.sh`. | ubu2 rm-disaster 2026-05-12; preempted in r2/r3 |
| **runpodctl CLI changed flags (2026-05)** вҖ” old `--imageName / --gpuType / --gpuCount / --containerDiskSize / --volumeSize / --secureCloud` вҶ’ new `--image / --gpu-id / --gpu-count / --container-disk-in-gb / --volume-in-gb / --cloud-type SECURE`. `--env` now takes a JSON object (`'{"KEY":"VAL"}'`), not repeatable `KEY=VAL`. Use `runpodctl gpu list` to get the exact `gpuId` string. | r3 (2026-05-12) |
| **`--env` does NOT propagate to non-interactive shells** вҖ” passing `--env HUGGING_FACE_HUB_TOKEN=...` makes it visible in `runpodctl pod get` and in interactive runpodctl-ssh, but **not** in `/etc/rp_environment` and **not** in `ssh <pod> 'bash <script>'`. **Workaround**: scp a `/workspace/.env` file (`export KEY=VAL` lines, mode 600) and `source` it at the top of `run_pod.sh`. Never put secrets on a command line via inline ssh. | r3 (2026-05-12) |
| **Endpoint discovery: `pod list` lacks IP/port** вҖ” `runpodctl pod list` returns id+name+status only. Use `runpodctl pod get <id>` for `.ssh.ip` + `.ssh.port` + `.ssh.ssh_command`. Port varies per launch (r2: 11159, r3: 13328). | r3 (2026-05-12) |
| **runpodctl `stop/remove` deprecation** вҖ” old `runpodctl stop pod <id>` / `runpodctl remove pod <id>` still work but warn. Use the new forms: `runpodctl pod stop <id>` / `runpodctl pod delete <id>`. | r3 (2026-05-12) |
| **RunPod injects RUNPOD_API_KEY into `/etc/rp_environment`** вҖ” every pod has the user's full API key in plaintext at that path. Don't grep/print the file in logs. (RunPod's design, not Claude's exposure вҖ” log-hygiene only.) | r3 (2026-05-12) |
| **Apples-to-apples scoring: do bf16 inference on the same H100** вҖ” don't NF4-quant on a 12 GB local GPU just to "save the pod minutes"; the bf16-vs-NF4 confound moves results several pp and pollutes the across-round ladder. | r2 fairness fix vs r29 |
| **Pull score files BEFORE stop+delete** вҖ” the pod's `/workspace/` is gone after `pod delete`. Either scp scores to ubu1 first, or rely on the HF push (the script-end `bench_subdir` upload) being the source-of-truth. | r3 (2026-05-12) вҖ” minor recover via HF |

**RunPod API surface (for v0.2.0-r10 SFT block):**

- **CLI** (`runpodctl` v0.16+): `pod {create, get, list, start, stop, delete, restart, reset, update}` В· `gpu list` В· `datacenter list` В· `user` В· `billing` В· `serverless / sls` В· `template / tpl` В· `hub` В· `model` В· `network-volume / nv` В· `registry / reg` В· `ssh` В· `send / receive`. Each `pod <verb>` accepts `-o json|yaml` for machine-readable output.
- **GraphQL API** (`https://api.runpod.io/graphql`): authenticated with the bearer token in `/etc/rp_environment::RUNPOD_API_KEY` (or your own from `console/user/settings`). Documented in RunPod docs; mutations cover create/start/stop/delete + pod-info queries. The Python SDK (`runpod`) wraps this. Use cases not covered by the CLI: filtering pods by tag, programmatic GPU-availability polling, building a dashboard.
- **Python SDK** (`pip install runpod`): wraps the GraphQL API; `runpod.api_key=$RUNPOD_API_KEY; runpod.create_pod(name=..., image_name=..., gpu_type_id=..., env={...})`. Note the SDK still uses `gpu_type_id` (the legacy field name); the CLI renamed it to `--gpu-id`.
- **Serverless endpoints**: separate from on-demand pods; deploy a model behind an HTTP API with autoscale-to-zero. Cost = per-second of GPU time. `runpodctl sls {create, list, delete}`. Use case for v1.0.0: serving the final 7B adapter via `dancinlab/hexa-forge-code-7b-...`.
- **Templates**: pre-baked image+env+volume configs you can launch with `--template-id`. Use case: encode the forge's "torch 2.4 + transformers 4.46.3 + peft 0.13.2 + trl 0.12.2 + accelerate 1.1.1" stack as a template so r5+ doesn't repeat the pip step.
- **Network volumes**: persistent storage across pod recreates. Currently unused by forge (all payload is scp'd per-run). Worth considering if HF download time becomes a bottleneck.

(Forge's own local-GPU runs already follow several of the original 12 rules: `setsid nohup`, sequential download-then-train, вүӨ2 checkpoints, full dep-verify. The r10 SFT block will encode all of these вҖ” plus the post-2026-05-12 CLI/API additions вҖ” as Q/A so the *model* knows them too.)

## 11. Container runtime вҖ” Docker (planned v0.3.1 SFT block)

Every GPU rental this forge uses (RunPod, Vast.ai) takes a **Docker image** as input. The model must know Docker well enough to (a) read/write Dockerfiles, (b) build custom images for the forge's stack, (c) debug image-runtime issues, (d) work with multi-arch builds (M-series Mac вҶ’ x86_64 GPU), and (e) understand the GPU runtime path.

| area | what the model must know |
| --- | --- |
| **Core CLI** | `docker {build, run, push, pull, exec, ps, logs, stop, kill, rm, image, container, volume, network, system}`. `docker build -t <name:tag> -f <Dockerfile> <ctx>`. `docker run --gpus all --rm -it -v <host>:<container> -p <host>:<container> -e KEY=VAL <image> <cmd>`. `docker ps -a` (all, including stopped). `docker exec -it <container> bash` (attach). `docker system prune -af --volumes` (reclaim space вҖ” DESTRUCTIVE). |

## 12. Self-aware delegation pattern вҖ” planned v0.4.0 architecture line

**The thesis.** A 7B is too small to know "all of programming." Instead of pretending, the forge teaches it to **recognize its competence boundary** and **delegate** out-of-domain requests to a larger model (Claude API) via a structured protocol. The 7B stays narrow + cheap + fast on hexa-canon work; everything else gets routed to a frontier LLM at run time. This is **routing-as-a-skill**, learned via SFT вҖ” not architecture (no MoE, no retrieval). Just a competence-classifier + a delegation-token emit + a runtime that closes the loop.

**Why not just train more knowledge in?** Three reasons. (1) Sub-7B can't memorize "all of Python + Rust + Go + ML internals" at useful fidelity. (2) The forge already showed (r30/r31) that adding even one domain (T5 HX-codes) crowds out others (T2/T8/T6). Adding "all of LLM internals" would shred hexa competence. (3) The owner already pays for Claude API; the 7B sitting on top of it is the right economic split вҖ” narrow specialist + frontier generalist.

**Why not just always delegate?** Three reasons. (1) Hexa-canon work doesn't need a frontier model вҖ” 7B at 80% on Mk.I is fine, fast, and free. (2) Latency: 50-100 ms local vs 2-5 s Claude API. (3) Offline + airgapped use cases need a local model that works without network.

### 12.A The competence boundary (what 7B does directly vs delegates)

| domain | answer directly? | trigger phrase the 7B should recognize | route if not direct |
| --- | --- | --- | --- |
| hexa code synthesis (T1 / T4) | вң… direct | "write a hexa function...", "implement L[N]...", "@grace...", "@implements..." | n/a |
| hexa diagnostics (T5 / T6 / T7) | вң… direct | "Which HX family covers...", "target triple for...", "stage S-N..." | n/a |
| hexa atlas annotation (T2 / T3) | вң… direct | "@implements(L[N])", "@discover(kind=L)", "@grace(HX...)..." | n/a |
| in-domain refusal (T8) | вң… direct ("refuse") | non-code prompts (poems / jokes / advice / multilingual) | n/a |
| 5-NL i18n (F1-F6) | вң… direct | English-canon diagnostics for non-English input | n/a |
| Apple/Swift small snippets | вң… direct | "write a Swift func...", small idiom Q | delegate if > 50 lines or framework-deep |
| Rust / Go / Python / TypeScript snippets | вҡ пёҸ delegate by default | any non-hexa code request | delegate |
| ML/LLM internals (transformers, attention, RAG, RL) | вқҢ delegate | "explain transformer self-attention", "how does prompt caching work" | delegate |
| general world knowledge / news / dates | вқҢ delegate | "what year was X", "who is Y" | delegate (or refuse if off-mission) |
| math derivations / proofs (beyond canon-equational) | вқҢ delegate | "prove that X = Y", "derive the closed form" | delegate |
| ambiguous / under-specified prompt | вҡ пёҸ ask-or-delegate | "do the thing" without object | delegate-with-clarify, or AskUser if interactive |
| security-sensitive / harmful | вң… direct ("refuse") | exfil / weapon / illegal | refuse вҖ” never delegate (don't shift liability) |

**Rule of thumb the model should internalize:** *"If the answer is in hexa-canon, answer. If it is a small idiom in a domain I know shallowly, answer with a confidence caveat. Otherwise emit a delegation token."* вҖ” That single sentence is the core of every Q/A pair in the v0.4.0 SFT block.

### 12.B Delegation token schema (the contract)

Single agreed token family the model emits; the runtime parses and acts. Proposal (subject to v0.4.0 spec):

```
<|delegate|>{"tool": "claude-api", "model": "claude-sonnet-4-6", "prompt": "<verbatim user prompt + any hexa context>", "max_tokens": 2048, "reason": "out-of-domain: <one-line>"}<|/delegate|>
```

- **`tool`** вҖ” exactly one of: `claude-api` (the default вҖ” calls the Anthropic API), `web-search` (if the runtime has it), `code-execution` (hexa-cc sandbox вҖ” for compile-back-and-fix loops). Forge v0.4.0 ships `claude-api` only.
- **`model`** вҖ” Claude model ID (`claude-opus-4-7` / `claude-sonnet-4-6` / `claude-haiku-4-5-20251001`). Default to `sonnet-4-6` for cost; let the model upgrade to `opus-4-7` if it says "this is a hard reasoning question". The 7B picks based on the prompt's reasoning depth (taught via SFT examples).
- **`prompt`** вҖ” the full text to send to Claude. The 7B should **enrich** it: prepend "You are answering a question for a small hexa-canon code-LLM that delegated this. Be concise; return code in hexa idiom if applicable." then the verbatim user prompt.
- **`max_tokens`** вҖ” budget (default 2048; the 7B picks 512 for short answers, up to 8192 for long code).
- **`reason`** вҖ” one-line explanation ("out-of-domain: Rust"), useful for telemetry + cost attribution.

**Why a JSON-inside-delimiters token, not a special vocab token?** (1) Works with stock Qwen2.5-Coder tokenizer (no new tokens to train). (2) The model already knows JSON well (T1 = 97.6%). (3) Easy to extend with new fields without retraining.

**Stop-token + format compatibility**: every SFT example that emits this token still ends with `<|endoftext|>` (the stop-token convention from r3). The runtime parser scans for `<|delegate|>` at any position in the generation.

### 12.C Runtime contract (what the harness / serving layer does)

This is the **runtime** side вҖ” the model emits; the runtime acts. Reference implementation lives in `tool/forge_runtime.py` (planned v0.4.0):

1. **Detect** `<|delegate|>...<|/delegate|>` in the model's emission. If absent вҶ’ pass the answer through to the user.
2. **Parse** the JSON. Validate `tool`/`model`/`prompt`/`max_tokens`. Reject malformed (return an error frame to the user; don't loop into the model).
3. **Authorize** вҖ” the runtime owns `ANTHROPIC_API_KEY` (from `~/core/secret/bin/secret get ANTHROPIC_API_KEY`); the **model never sees the key**. This is the safety boundary.
4. **Budget check** вҖ” per-conversation cost cap + per-day spend cap. If exceeded вҶ’ return a budget-exhausted frame (the model learns to handle this gracefully).
5. **Call Claude API** with prompt caching enabled (the runtime caches the system prefix; per В§9.B 5-min TTL).
6. **Inject the response** back into the model's context as a synthetic tool-result token: `<|delegate-result|>{...claude response text...}<|/delegate-result|>` and re-run the model from there. The model then writes the user-facing wrap-up using Claude's content as ground truth.
7. **Cap iterations** вҖ” `max_delegations_per_turn = 3` (the model's SFT teaches "if you've already delegated once for this turn, don't delegate again вҖ” answer with what you have, or refuse").
8. **Telemetry** вҖ” every delegation logs `{ts, model, tokens_in, tokens_out, cost_usd, reason, user_id}` to `state/delegation_log.jsonl` for cost attribution.

### 12.D Failure modes the model must handle (taught via SFT)

| failure | runtime emits | model's correct response |
| --- | --- | --- |
| Claude API timeout / 5xx | `<|delegate-result|>{"error":"upstream_timeout"}<|/delegate-result|>` | "I tried to delegate this to a larger model but the upstream is unreachable. Best I can offer locally: [partial answer or honest unknown]." |
| Budget exhausted | `<|delegate-result|>{"error":"budget_exhausted"}<|/delegate-result|>` | "This question is outside my hexa-canon competence and the delegation budget for this conversation is exhausted. Please rephrase as a hexa question, or retry later." |
| Network down (offline pod) | runtime detects, returns `{"error":"offline"}` | "I am running offline; I can only answer hexa-canon questions directly. For non-hexa questions, please retry on a networked instance." |
| Malformed delegation (model's own bug) | runtime parse fail | the model should **not** see this вҖ” runtime returns a hard error to the user. SFT must teach the model to emit *only* well-formed JSON; this is a never-event. |
| Delegated answer is a refusal (Claude refused) | `<|delegate-result|>{"text":"I can't help with that"}<|/delegate-result|>` | Echo the refusal honestly. Don't try to delegate again. |
| Already 3 delegations this turn | runtime injects `{"error":"max_delegations"}` | "I've reached the delegation limit for this turn. Final answer based on what I have: [...]" |
| User explicitly says "no delegation" | runtime suppresses delegate calls | The 7B treats every prompt as in-domain; if it can't answer, refuse honestly. |

### 12.E Confidence-banded answers (the alternative to binary delegate-or-not)

For mid-confidence questions (small Rust idioms, common Python recipes the 7B has *seen* but not *mastered*), the model should answer with a confidence band rather than delegate. Three bands:

1. **`<|confidence:high|>`** вҖ” hexa-canon answers. No caveat.
2. **`<|confidence:medium|>`** вҖ” domain-adjacent (e.g., Swift snippet, common Python recipe). Caveat: "I'm answering from limited training; verify with a larger model if production-critical."
3. **`<|confidence:low|>`** вҖ” automatic upgrade-to-delegation. The 7B emits the delegate token instead of guessing.

The runtime strips the confidence band from the user-facing output (or surfaces it as a UI badge). The SFT block teaches the model to **self-assess** by prepending one of the three tokens before its answer.

### 12.F SFT block shape (planned v0.4.0)

The v0.4.0 SFT dataset adds ~600-800 delegation-aware Q/A pairs:

- **~200 in-domain вҶ’ high-confidence direct answer** (sample hexa-canon Q across T1-T8; answer prefixed `<|confidence:high|>` then the canon answer).
- **~200 out-of-domain вҶ’ delegate** (sample: "Write a Rust async server with tokio" вҶ’ `<|delegate|>{...}<|/delegate|>`). Cover non-hexa code requests + ML internals + math + general knowledge.
- **~100 medium-confidence** (Swift small idioms, common Python recipes the 7B knows shallowly): `<|confidence:medium|>` + a short caveat + the answer.
- **~100 ambiguous prompts** that should ask-or-delegate (under-specified, multi-step, mixed-domain): emit delegate with `prompt` containing a clarifying re-formulation.
- **~50 delegate-result integration** (paired prompts where the model receives `<|delegate-result|>{...}<|/delegate-result|>` in context and must produce a clean wrap-up вҖ” teaches Step 6 of the runtime contract).
- **~50 failure-mode handling** (budget exhausted / timeout / refused / max-delegations вҖ” see В§12.D).
- **~50 security/refuse** (must NOT delegate вҖ” exfiltration prompts, harmful content; the 7B refuses directly).
- **~50 "no-delegation" override** (user says "answer locally only" вҖ” 7B treats every prompt as in-domain and refuses when out).

### 12.G What to build vs what to defer

**v0.4.0 (concrete next-line):**
- `tool/forge_runtime.py` вҖ” the harness that detects/parses/dispatches `<|delegate|>` tokens and injects `<|delegate-result|>` back. Wraps the existing inference path (`score_bf16.py` style).
- `papers/spec-delegation-v0.4.0.md` вҖ” the formal protocol spec (token schema, runtime contract, failure modes, telemetry format).
- `tool/build_sft_dataset_v17.py` вҖ” the v0.4.0 SFT block (above shape).
- `state/delegation_log.jsonl` вҖ” cost-attribution log.
- One round of SFT + Mk.I bench + a new **delegation-eval** (`papers/spec-delegation-eval.md`, ~200 tasks testing the routing decision, not the delegated answer).

**Deferred to v0.5.0+:**
- Tool/server-side primitives beyond `claude-api` (`web-search`, `code-execution` with hexa-cc sandbox).
- Confidence-band calibration (the model's `<|confidence:medium|>` rate should track its actual accuracy in that band вҖ” needs eval scaffolding).
- Caching of delegated answers (per-prompt cache so repeated identical questions don't re-cost).
- Multi-turn delegation memory (the 7B remembers what it delegated in turn 1 and skips re-delegating in turn 3).
- An offline-graceful fallback: a tiny local "general-knowledge" 1.5B that answers low-stakes non-hexa questions when network is down. Probably not worth the complexity.

### 12.H Cost model (rough)

- **Per delegation**: ~500 input tokens (the model's prompt + system prefix) + ~1000 output tokens вүҲ $0.003 with Sonnet 4.6 + caching, ~$0.012 with Opus 4.7. The 7B's job is to use Sonnet by default and only escalate to Opus when "this is hard reasoning".
- **At 100 conversations/day with avg 1 delegation each**: ~$0.30-$1.20/day for the delegation tier. Negligible vs the SFT round cost.
- **Budget caps in runtime** prevent runaway loops (e.g., a buggy SFT-fine-tune that delegates everything вҶ’ bounded by the per-conversation cap).

## 14. Wilson вҖ” hexa-native AI coding agent (the in-house ROUTING target)

**The fourth delegation target, qualitatively different from Claude/OpenAI/Gemini.** Wilson is the owner's **self-hosted agent harness**, written entirely in hexa-lang (`~/core/wilson`) вҖ” same role as Claude Code / Codex CLI / Gemini CLI but **plugin-first + hexa-native + LLM-agnostic** (it shells out to whichever provider plugin is registered). For the 7B this matters two ways:

1. **As a *coding target*** вҖ” the 7B should be able to write Wilson plugins on demand (hexa-canon synthesis target, see В§2 hexa-canon competence).
2. **As a *routing target*** вҖ” `<|delegate|>{"tool":"wilson-rpc", ...}` invokes a Wilson swarm cell. The forge runtime can hand off heavy in-house work (eval scoring, multi-file refactors, batch SFT-data generation) to a local Wilson process instead of paying Claude/OpenAI/Gemini API for it.

### 14.A What Wilson is (the architecture in one screen)

### 14.D Delegation schema extension вҖ” `tool: "wilson-rpc"`

Extending В§12.B's delegation token to include Wilson:

```
<|delegate|>{
  "tool": "wilson-rpc",
  "bundle": "_bundle_cell",
  "cell_count": 4,
  "payload": {
    "task": "score-mk1",
    "manifest": "/workspace/manifest-mk1.jsonl",
    "adapter": "dancinlab/hexa-forge-code-7b-...-v0.3.0-r4",
    "shard": "0/4"
  },
  "max_seconds": 1800,
  "reason": "in-house batch work вҖ” no LLM call needed"
}<|/delegate|>
```

- **`tool: "wilson-rpc"`** вҖ” runtime spawns `cell_count` Wilson processes in `--harness rpc` mode, each receives the shard of work via stdin JSONL, returns results via stdout JSONL.
- **`bundle`** вҖ” usually `_bundle_cell` (the minimal swarm bundle: `harness-rpc + tool-core + provider-anthropic`). Default `_bundle` for full agent capability.
- **`payload`** вҖ” opaque task descriptor. The receiving Wilson cell unpacks via the active provider/tools.
- **`max_seconds`** вҖ” hard timeout (runtime kills cells past this).
- **`reason`** вҖ” telemetry attribution.

**When to pick `wilson-rpc` over `claude-api`:**
- Mechanical work that fits 7B's tools (file ops, git ops, parse-check, regex transforms) вҖ” **wilson-rpc** ($/call вүҲ 0 if local, only pays for the *inner* provider if needed)
- Reasoning-heavy / world-knowledge / multi-language synthesis вҖ” **claude-api / openai-api / gemini-api**

**The cost-routing ladder (cheapest вҶ’ most expensive):**

1. **`wilson-cli`** вҖ” single deterministic tool call (no LLM, no agent loop). E.g. `wilson tool read /etc/hosts`, `wilson task add ...`, `wilson memory list`. **Cost вүҲ 0** (just a subprocess).
2. **`wilson-rpc`** вҖ” multi-step in-house batch via swarm cells (agent loop runs locally; inner provider only fires if a cell decides it needs LLM). **Cost = LLM only on the cells that escalate**, usually a fraction of doing the whole thing on Claude.
3. **`claude-api` / `openai-api` / `gemini-api`** вҖ” frontier reasoning / external knowledge / non-hexa code synthesis. Full LLM cost.

Route to the **lowest tier that can answer the prompt as-given**. The 7B's routing intelligence (taught via SFT) is mostly about recognizing which tier the request lives in.

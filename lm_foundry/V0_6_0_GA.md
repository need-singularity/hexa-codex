## 1. What v0.6.0 IS

v0.6.0 is a **production-ready forge code-LLM platform** consisting of:

1. **Pure-specialist 7B model** (frozen at r39) — hexa-canon programming
   LLM trained from Qwen2.5-Coder-7B + LoRA r=64 via SFT (r1-r34) →
   Phase-A manifest fixes (r33+r37+r38) → compile-feedback RL via GRPO
   (Lever 4 — T4 enum 55→100%) → T3 quote-fragility patch (r39 — T3
   58.8→100%). **94.29% Mk.I strict / 96% 5-NL i18n**, both unchanged
   since the GA mark.

2. **Pre-7B keyword classifier** (`tool/classify_prompt.py`, ~470 LOC) —
   stage-based regex + signal-weighted disambiguation. Labels prompts as
   `hexa | ood | refuse`. On the 300-task DLG-mk0 manifest: **98.33%
   overall**, 100% on in-domain / mid-conf / refuse / ambiguous / long-
   context categories, 94.9% on ood-delegate.

3. **Per-vendor tier selector** (`tool/select_vendor_tier.py`, ~225 LOC) —
   pure function mapping classifier signals to vendor + model + max-
   tokens. 6-step priority cascade: longctx (gemini-2.5-pro) → ml-
   comparison demotion (claude-sonnet) → reason-algo (openai o4-mini) →
   reason-deep (claude-opus) → struct (gpt-5-mini) → general (claude-
   sonnet). On the 77 must_delegate tasks: **100% tier_match**, **99.26%
   tool_match** (cross-vendor tier-equivalence applied).

4. **Real 3-vendor SDK dispatch** (`tool/forge_runtime.py`,
   ~2200 LOC) — anthropic + openai + google.genai with proper error
   mapping (auth_fail / upstream_timeout / upstream_5xx / upstream_quota
   / schema_violation / redaction_block), PII redaction (8 classes),
   per-prompt + per-conversation budget caps, latency-aware filler
   tokens, idempotent telemetry.

5. **Multi-tier persistent caching**:
   - In-memory dict (LRU, 5-min TTL) for hot path
   - JSONL file-backed (single-process restart-persistence)
   - SQLite WAL (multi-process safe; concurrent reads + serialized writes)
   - Cron-friendly `forge_vacuum` for periodic cleanup

6. **Multi-turn delegation memory** — per-conv buffer with max_turns
   cap, 3 dispatch modes: single-turn (default) / string preamble (r57)
   / native vendor messages (r59).

7. **Calibrated confidence emission** (`_emit_conf` helper, r54) —
   Brier **0.0242 EXCELLENT** / ECE **0.0461 GOOD** on the 300-task
   DLG-mk0 manifest (label dispatch unchanged by construction).

8. **Production observability** (`tool/forge_audit.py`, ~660 LOC):
   - Cache hit rate / cost-saved estimate
   - Vendor + tier distribution
   - Error code breakdown
   - Latency p50/p95/p99
   - Cost attribution + top-10 expensive turns
   - Health gate alerts (cache_hit_min / error_rate_max / cost_day_max
     → exit code 2 on breach)
   - 3 output formats (text / json / csv)

9. **Operational tooling**:
   - `tool/run_all_smoke.py` — 7-step unified verifier, 4.50s wall time
   - `tool/perf_bench.py` — p50 556μs / p99 1.66ms for classifier+selector
   - `tool/forge_vacuum.py` — SQLite cleanup CLI (expire + LRU cap + retention + VACUUM)
   - `tool/forge_keys.py` — vendor API key management (status / add / remove / test)
   - `tool/smoke_e2e_r53.py` — end-to-end production smoke (24 prompts, 2 vendors verified)
   - `tool/bench_anthropic_cross_turn.py` — A/B cache marker measurement

10. **Specification + runbook**:
    - [`ORCHESTRATION.md`](ORCHESTRATION.md) — 15-section spec (the "what")
    - [`OPERATIONS.md`](OPERATIONS.md) — 10-section runbook (the "what you do")
    - [`ROADMAP.md`](ROADMAP.md) — 28-round narrative (the "how we got here")

## 2. What v0.6.0 IS NOT

Honest scope-cut list. These are **not GA in v0.6.0** and operators
should plan around them:

### 2.1 User-action gating

- **OpenAI API key** is not provisioned in the secret store at GA
  release time. Reason-algo (o4-mini) and struct (gpt-5-mini) tier
  routes will return `auth_fail` until operator adds the key via
  `tool/forge_keys.py add openai`. Until then, prompts routed to those
  tiers fall through to the user-facing "auth not configured" message.
- **Gemini paid-tier project** — the registered `gemini.api_key` is
  from a free-tier project; `gemini-2.5-pro` returns `upstream_quota`.
  `gemini-2.5-flash` works on free tier. For longctx route to actually
  serve responses (not just map the quota error), operator needs to
  swap to a paid-project key via `tool/forge_keys.py add gemini`.

### 2.2 Measurement gaps

- **Anthropic cross-turn cache savings** for opus and haiku are
  empirically ZERO (r66). Only sonnet auto-caches at ~90% savings on
  the cached portion. **Do not predict cost savings from cross-turn
  caching for opus/haiku in production planning.**
- **Multi-process SQLite WAL contention** at >1000 writes/sec is
  untested. The 10-second connection timeout is documented; for higher
  throughput, see OPERATIONS.md §5 (batch writes or move to Postgres /
  managed Redis).
- **Production telemetry baselines** are not yet established. The
  300-task DLG-mk0 manifest is held-out but synthetic; the
  workload-dependent cache hit rate / error rate / cost-per-day
  numbers in OPERATIONS.md §4 are conservative defaults, not measured
  baselines. **Production rollout should run `forge_audit` for ≥1
  week before tuning health gate thresholds.**

### 2.3 Architectural deferrals (v0.7+)

- **Specialist ceiling improvement** — Lever 5+ (full-FT / larger LoRA
  rank / more SFT data) OR routing-LoRA as a separate weight artifact.
  Both are GPU-bound and were deferred at the v0.5.x architectural
  pivot (r44).
- **Schema migration** — `SCHEMA_VERSION = 1` is detection-only
  (warns on mismatch via `PRAGMA user_version`). A real migration story
  requires either versioned filenames (`forge.v2.sqlite3`) or an
  explicit migration script.
- **SQLite incremental vacuum** — would let `VACUUM` run concurrently
  with reads. Requires schema migration at DB creation time.
- **Multi-process shared cache via real network DB** (Postgres /
  managed Redis / DynamoDB) — the JSONL/SQLite WAL backends are
  local-disk only; distributed deployments need a network DB layer.
- **Connection pooling** at very high throughput (>10K writes/sec) —
  the current per-runtime single connection is fine to ~1K writes/sec
  on modern SSD. Higher throughput would need batching or a real pool.
- **Vendor-native message-list cache-marker structure for opus/haiku**
  — r66 showed sonnet auto-caches via system marker but opus/haiku
  don't. Closing this gap (if cost savings prove valuable) requires
  per-model investigation of marker placement.

## 4. Deployment recipe

Minimal v0.6.0 production deployment (single-process):

```python
from pathlib import Path
from forge_runtime import ForgeRuntime, ForgeRuntimeConfig

cfg = ForgeRuntimeConfig.from_env(
    vendor_cache_path=Path("/var/lib/forge/cache.jsonl"),
    multi_turn_memory_enabled=True,
    conv_history_path=Path("/var/lib/forge/conv.jsonl"),
)
rt = ForgeRuntime(cfg)
```

Multi-process production deployment (load-balanced):

```python
cfg = ForgeRuntimeConfig.from_env(
    vendor_cache_enabled=True,
    multi_turn_memory_enabled=True,
    multi_turn_memory_auto_prepend=True,
    multi_turn_memory_native_messages=True,
    forge_db_path=Path("/var/lib/forge/forge.sqlite3"),  # SQLite WAL
)
rt = ForgeRuntime(cfg)
```

Daily maintenance cron (see OPERATIONS.md §2 for full template):

```cron
0 3 * * * forge python3 /opt/forge/tool/forge_vacuum.py --db /var/lib/forge/forge.sqlite3 --keep-recent 4096 --conv-days 30
30 3 * * * forge python3 /opt/forge/tool/forge_audit.py --since-hours 24 --alert-cache-hit-min 0.20 --alert-error-rate-max 0.05 --alert-cost-day-max 50.00 || mail -s "forge degraded" oncall@example.com
```

Pre-deploy verification:

```bash
python3 tool/run_all_smoke.py          # 7-step unified verification, 4.50s
python3 tool/forge_keys.py status      # verify all 3 vendor keys present
python3 tool/forge_keys.py test all    # live-verify all 3 keys
```

## 6. What changes in v0.7+

The v0.7.0 line will focus on items that require either GPU compute or
architectural shifts that v0.6.0 left for later:

1. **Specialist ceiling improvement** — Lever 5+ or routing-LoRA. Both
   GPU-bound, both rewrite the r39 GA adapter. Expected cost: ~\$5-15
   per training round depending on hardware.

2. **Network DB backend** — Postgres / managed Redis / DynamoDB for
   true multi-host deployments. Schema migration story will need to
   be solved before this lands.

3. **Auto-retry with backoff** — `upstream_5xx` and `upstream_quota`
   currently surface to the user-facing message; v0.7.0 candidate is
   auto-retry with exponential backoff in the runtime, capped at
   N retries with bounded delay.

4. **Bio verb activation** — `LEARNING_BIO.md` exists but no training
   has happened. Paired with `dancinlab/hexa-bio` data repo. GPU-bound.

5. **gemini-2.5-pro paid-tier engagement** — once operator swaps to
   paid-project key, longctx route gains real ROI measurement; the
   r48 quota mapping is verified but actual long-doc answer quality
   is not yet measured.

## 8. Honest closing notes

This v0.6.0 GA mark is **conservative**:

- The specialist quality (94.29% Mk.I) is unchanged from r39 and has
  been validated across the entire v0.5.x line by construction (no
  re-training).
- The orchestration runtime has 60+ test gates with zero regression
  across r44-r66.
- Every honest finding (r43 zombie loss, r62 redundant marker, r66
  per-model cache variance, gemini free-tier limit) is documented in
  the round narrative and surfaced in §15 honesty caveats.
- No claim is made that this is the BEST possible code-LLM stack —
  only that what was built has been verified and the gaps are
  enumerated above.

The honest answer to "is v0.6.0 production-ready?" is: **yes for
single-or-multi-process anthropic-routed deployments; partial for
openai/gemini until operator provisions keys; the specialist itself
is frozen at r39 quality and any improvement requires v0.7+ GPU work.**
